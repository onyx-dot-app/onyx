# Mobile file-upload module restructure — Research

## Requirement

Restructure the mobile app's file-handling into a clean, reliable, **file-keyed** module — one core hook plus two lean lens hooks (message attachments, project files) over a single zustand store (not React context) — that survives the states mobile throws at it, and fix the three P2 bugs deferred from PR 8.

## Clarifications (Q&A)

- **Q: Why a store instead of context here?** Mobile's chat UI is one persistent `ChatSurface` overlay that morphs across routes and never remounts; upload writes come from detached async callbacks that outlive their component; and progress ticks at high frequency. All three make context the wrong tool (Provider-scoped, re-renders all consumers, no clean detached-writer path) and a store-outside-React the right one. Web *can* use `ProjectsContext` because its Provider sits above conventional mount/unmount routes.
- **Q: Why does today's `uploadStore` feel project-smeared?** It is keyed `byProject: Map<number, ProjectUploads>`, baking "an upload belongs to a project" into the upload's *identity*. Per-message attachments (no project) can't use it, so PR 8 forked a second local-state layer — two parallel file systems. Fix: key by `fileId`; project/message become *links* layered on top.
- **Q: How resilient must an in-flight upload be? (owner decision)** → **Survive navigation only (in-memory store).** Lost on app kill/restart, but the file is safe server-side once bytes land and status re-polls on return. Not persisting to MMKV, not doing true background transfer — but **bake in two seams** so a future background PR is bounded: a `UploadTransport` interface and server-as-source-of-truth reconciliation.
- **Q: What happens on minimize?** Process usually stays alive → store intact, UI restores exactly. The native foreground uploader may pause under iOS suspension and resume on return; the redesign re-polls `/statuses` on `AppState` "active". Only an OS process-kill drops the *unsent draft* (uploaded files still survive server-side).
- **Q: Does later background upload need large changes?** No — the store, hooks, reconcile, and UI don't move. Later work is confined to a second `UploadTransport` impl (native module + config plugin + prebuild) plus a small MMKV persist slice, *provided* the two seams above are built now.

## Web reference (the multi-list ProjectsContext is the ANTI-PATTERN, not structure to copy)

- `web/src/providers/ProjectsContext.tsx` is a single provider mounted **above the routes** (`app/app/layout.tsx`) — that's why web survives navigation.
- It holds **five overlapping arrays of full file objects** (`recentFiles`, `allRecentFiles`, `allCurrentProjectFiles`, `currentMessageFiles`, `currentProjectDetails.files`) + `projectToUploadFilesMapRef` + `trackedUploadIds`, hand-reconciled: a 3s poll writes into **three** of them and a ~23-dep `useMemo` splices link/unlink/delete (lines 393-421, 579-655). This is exactly the duplication the normalized store replaces — web keys by file identity but then **denormalizes into many lists that must be kept in sync by hand**. Treat it as the failure mode to avoid, not a shape to port: mobile keeps ONE record per file and derives every surface as an id-list over it.
- `beginUpload(files, projectId?, onSuccess, onFailure)` (:347) creates optimistic `temp_id` files, uploads, swaps `temp_id → server`, then polls `/statuses` every 3s. link/unlink are optimistic with rollback (:748-785).

### Backend facts (verified)

1. **Recent = the full library.** `GET /user/files/recent` returns the user's whole file pool — no `LIMIT`, no time-bound `WHERE`; excludes only FAILED/DELETING; `ORDER BY last_accessed_at DESC` (`backend/onyx/server/manage/users.py:1305`). "Recent" is just the pool, sorted.
2. **A project file is a pure many-to-many association.** `Project__UserFile` join (`backend/onyx/db/models.py:5069`); link/unlink add/remove a join row. A project file is the SAME standalone `UserFile` by id — a subset of the same pool as recent.
3. **The backend models file = identity, membership = association.** `UserFileSnapshot.project_id` is hardcoded `None` (`backend/onyx/server/features/projects/models.py:38`). A normalized client cache mirrors the backend's own shape — it is not speculative generality.
4. **Web is the counter-example.** Its `ProjectsContext` keeps 5 duplicated arrays of full file objects reconciled by hand — the multi-list approach the normalized store removes.
5. **Offline: no regression.** `projects` and `recent-files` are ALREADY PII-excluded from Query's MMKV persistence (`mobile/src/api/query/client.ts`), so committed files were NEVER on disk. Moving them into an in-memory store loses nothing, and the future `persist` slice is identical.

## Industry best practices

- **Normalized `byId` dictionary** keyed by a stable id is standard whenever an upload is shown/mutated from multiple screens or outlives its component — exactly this app. Prefer a plain `Record` over `Map` (cheap spread; if `Map`, always `new Map(...)`, never mutate in place, or the UI freezes while progress "updates"). — redux.js.org normalizing-state-shape; vovk.dev/realtime-ui/state
- **Zustand perf:** atomic per-file selectors (`s => s.byId[id].progress`); never return a fresh object/array from a selector (reference-equality defeat → re-render storm); `useShallow` for grouped reads; `subscribeWithSelector` transient updates to drive a high-frequency progress bar imperatively without React re-renders. — github.com/pmndrs/zustand; dev.to zustand perf
- **Store (not Context)** for state written from detached async callbacks, surviving unmount, updating at high frequency — community consensus. — adamhinckley.com zustand-vs-context; dev.to context-vs-zustand
- **expo-file-system:** `File().upload`/`createUploadTask` give `onProgress {bytesSent,totalBytes}` + `cancel()`; background `sessionType` survives native suspension but the JS task/promise is NOT restored after termination; uploads survive navigation only if the task lives outside the component (→ store). — docs.expo.dev/sdk/filesystem
- **Optimistic:** temp-id→server-id swap, server WINS on merge, keep optimistic distinguishable (`isPending`/status), and a reconciliation pass (poll/refetch) must recover "succeeded-but-callback-dropped" so nothing strands in "uploading" forever. — 7tech react-optimistic-ui-blueprint; sitepoint useOptimistic patterns
- **Monotonic epoch/request-id guard** (capture `id=++counter` at run start; `if (id !== current) return` in the callback) is the ONLY correctness fix for late/out-of-order writes — debounce is not. — frontendatlas js-async-race-conditions
- **One file, many contexts:** file is the identity entity (`byId`); each context (project link, message attachment) is a lightweight reference `{ fileId, ...contextMeta }`, never a duplicated copy. — vovk.dev; redux normalizing-state-shape

## Approaches

All three honor the owner's fixed decisions (zustand store not context; key by `fileId`; core hook + two lens hooks; in-memory-only v1; a `UploadTransport` seam + server-as-truth reconcile so background is a bounded later add). **They diverge on exactly one axis: the store's scope — where committed project files and the per-message draft live.** All three fix the three bugs the same structural way (draft lives in the store and survives `submit`, so it clears only on an *accepted* send; a monotonic epoch guard replaces the conversation-keyed guard; per-file errors replace the floating banner; `resolveMaxUploadMb` centralizes BUG2 in one finite-fallback helper) and all three land the `BearerImage` rider as an independent commit.

### Approach A — Simplicity-First: Ephemeral-only file-keyed store

**Store scope:** the store owns **only the ephemeral upload lifecycle** — `files: Record<fileId, UploadEntry>` where `UploadEntry = { file, progress, error, epoch }`, plus two id-list link maps `drafts[draftKey] → fileId[]` and `projectUploads[projectId] → fileId[]` (in-flight uploads only), plus `contextErrors`. **Committed project files stay in the existing TanStack Query `userProject` cache** — the store never shadows the server cache; it holds only what Query can't (rows with no server id yet, or draft-only files).

- **Hooks:** a core `useFileUploads` (owns transport + size precheck + `/statuses` poll + AppState re-poll + epoch) that holds no React state; two thin lenses. `useMessageAttachments` reads `drafts[key]`; `useProjectFiles` prepends `projectUploads[id]` optimistic rows onto the Query `committedFiles` and keeps link/unlink → `invalidateQueries` exactly as today.
- **Selectors:** atomic `useFileProgress(id)` (hot path, one card re-renders per tick); `useDraftFileIds`/`useProjectUploadIds` (id array, changes only on add/remove/reconcile); `useFilesByIds` via `useShallow` over `.file` refs (stable across ticks because `setProgress` spreads the entry but not `entry.file`).
- **Reset contract:** replaces destruction with *selection* — the lens just selects `drafts[key]`; chat B never sees chat A's draft; a draft drops only on `consumeDraft` (accepted send, removes only entries it *solely* owns) or explicit remove. No reset effect, so BUG3's ghost race is structurally impossible.
- **Trade-offs:** smallest diff (`byProject → byId` + two link maps; delete the PR8 forked local-state layer; net LOC down); zero store↔Query duplication or sync risk; Query stays the proven `serverUrl`-keyed/MMKV/focus-refetch cache. **Sacrifices:** two reconcile *tails* (message = in-store temp→server re-key; project = drop optimistic + Query invalidate), unified at the engine entry but divergent at the end; the store can't answer "everything about file X, committed or not" in one selector; a file that is both project-linked and message-attached would be two rows in two systems (not a need today).
- **Risks:** never-sent drafts accumulate in memory (bounded, wiped on restart); `reconcile` must re-key `files` + every referencing link in one `set`; selector discipline (a stray whole-`files` select reintroduces the tick storm); `setProgress` spreads the `files` Record each tick (O(#files), negligible for a handful).

### Approach B — Scalability/Robustness-First: Task-centric upload-manager store

**Store scope:** an upload is a **first-class task**. `tasksById: Record<taskId, UploadTask>` where `UploadTask = { taskId, clientId, fileKey, target, epoch, status, progress, error, transport }` holds progress/epoch/cancel; a separate `filesById: Record<clientId, FileRecord>` holds canonical reconciled records; `projectLinks`/`drafts` are `clientId[]` lists. `clientId` is assigned once and **never moves** on temp→server swap (only the `ProjectFile` view's ids fill in). Non-serializable cancel handles live in a module-level `runtimeHandles` side table, out of state.

- **One reconcile reducer** `reconcile(serverFiles, epoch?)` fed by all three producers (transport result + 3s poll + AppState resume re-poll), server wins, one code path.
- **Background-ready:** `UploadTransport` injected via `configureUploadTransport`; a null `UploadReconciler` component mounted in `_layout.tsx` fires `reconcileOnResume()` on foreground; state is all JSON-serializable `Record`s so option-2 MMKV is later just `persist({ partialize })` — no reshape.
- **Trade-offs:** the single reconcile invariant + uniform epoch guard + normalized identity are the strongest correctness story, and the transport/resume seams make both the background PR *and* dropped-callback "orphan-success" recovery fall out. **Sacrifices:** the largest surface (task entity + transport + reconciler + resume) for an in-memory-only v1 where much of it is dormant; two representations of a file's live state (`record.status` + `task.progress`) the card must read together; needs a GC.
- **Sharpest risk:** orphan recovery for **draft (`projectId=null`) uploads** depends on `/user/files/recent` echoing the `temp_id` — **unverified against the backend**; if it doesn't, a dropped-callback draft upload can't be matched on resume and stays `UPLOADING` until re-picked. Also: migrating `api/files/upload.ts` from `File().upload` to `createUploadTask` (for a real `cancel`) changes the error/response shape and must be re-validated.

### Approach C — Normalized single-file-cache SSOT (Query fetches-and-hydrates, never renders)

**Store scope:** `userFileStore.filesById: Record<fileId, FileRecord>` is **the single source of truth for EVERY file record — uploads, draft attachments, project files, AND recent/library files**. Relationships are id-only: `projectFileIds: Record<projectId, string[]>` (a project's committed membership, ordered) and the composer draft's `fileIds`; the recent picker holds `recentIds: string[]`. **TanStack Query is demoted to a fetch-and-hydrate LOADER** — it still fetches (retry / dedup / focus-refetch / lazy `enabled`), but its results HYDRATE the store on success (`hydrateProject`, `hydrateRecent`) and are **NEVER rendered for files**. One source, several loaders. A React-free upload driver owns orchestration.

- **Cleanest "file is identity, context is a link":** exactly one record per file; the single `UploadReconciler` poll updates *every* surface showing that file at once; a future global file-library screen or shared `@onyx-ai/shared` registry is a new selector over existing storage; background transport is an injected swap.
- **The dual-source drift the original C feared is resolved by one rule — "Query never renders files".** There is only ever one rendered source (the store); Query is a pure loader whose only effect is to hydrate it. The earlier C's objections are answered by the new backend/web evidence (see *Backend facts*): the "speculative cross-context file" objection is void because the backend itself models file = identity + membership = association (`project_id` hardcoded `None`), so a normalized cache is the *faithful mirror*, not speculative generality; the "fights convention" objection is void because Query keeps its fetch mechanics — it just stops being the render source for files.
- **Trade-offs:** best identity model and best future-surface extensibility, with drift removed. **Sacrifices:** the largest migration; the store now owns committed-file lifecycle (bounded by library size, in-memory, wiped on restart); needs bounded retention (dangling refs are safe — `useFilesByIds` filters missing ids — so an explicit `gc()` is optional/LRU-later).
- **Residual concern:** membership must stay server-authoritative — handled by hydrate on fetch + optimistic add/remove on link/unlink + a project-details refetch that re-hydrates. A `hydrate` and an in-flight temp→server swap must not clobber each other (`reconcile` owns the swap by `temp_id`; hydrate upserts records without discarding an unreconciled optimistic row).

## Cross-comparison

- **Migration size:** A (smallest, net LOC down) < B (task machine + reconciler) ≈ C (single store + hydrate + driver).
- **Duplication / drift risk:** A none (Query owns committed, store owns ephemeral) · B minimal (Query owns committed; store optimistic-only) · C **none — ONE source**: the store is the only render source and the drift the original C feared is removed by the rule "Query never renders files" (Query only hydrates).
- **Faithfulness to the backend model:** C highest — the backend's own file = identity + membership = association shape (`project_id` hardcoded `None`, `Project__UserFile` join, recent = full pool) makes the normalized single-cache the *mirror* of the server, not speculative generality · A/B keep a Query/store split that re-splits what the backend keeps as one pool.
- **Correctness ceiling under concurrency/background:** B and C both carry one reconcile reducer (server wins) · A has two reconcile tails.
- **Concrete new risk introduced:** A ~none · B a backend `temp_id`-echo dependency + a `createUploadTask` transport migration · C a hydrate↔reconcile ordering rule (`reconcile` owns the temp→server swap; hydrate upserts without clobbering an unreconciled optimistic row) — *no* dual-source drift, since Query never renders.
- **Fixes all three bugs structurally:** all three (identical mechanism).
- **Background PR bounded later:** all three (all wire the `UploadTransport` + server-as-truth seams); state is JSON-shaped `Record`s so option-2 MMKV is a `partialize`, not a reshape.
- **The multi-list duplication C removes** — 5 hand-reconciled arrays of full file objects — **is the exact anti-pattern web's `ProjectsContext` demonstrates today.**

## Chosen approach

**Approach C (refined) — normalized single-file-cache SSOT** (one id-keyed `filesById` store for EVERY file; Query fetches-and-hydrates, never renders). **This REVERSES the earlier B choice**, and the reversal is deliberate — new backend/web research showed the drift fear that sank the original C applied only to the *half-measure* (a registry that shadows a still-rendered Query cache). Committing fully removes it: with the rule **"Query never renders files,"** there is exactly one rendered source (the store) and Query is a pure loader whose only job is to hydrate it — no dual-source render, so no drift. The evidence makes C the *faithful* model, not the speculative one: recent is the **full library** (no LIMIT), a project file is the **same `UserFile` by id** through a `Project__UserFile` join, and the backend hardcodes `project_id=None` — file = identity, membership = association, which the normalized cache mirrors exactly. Web's `ProjectsContext` proves the alternative rots (5 hand-reconciled arrays, a 3s poll writing three of them, a 23-dep splice), and offline has **no regression** (projects + recent were already PII-excluded from MMKV, so committed files were never on disk). B's task machine and A's Query/store split both re-divide what the backend keeps as one pool; C keeps it one. C inherits B's strengths that survive the pivot — one reconcile reducer (server wins), the `UploadTransport` seam, an AppState resume reconciler (now the *single* `UploadReconciler` over the whole store), and JSON-shaped state for a later `partialize`.

**Load-bearing items to verify as C is built:** (1) **membership stays server-authoritative** — link/unlink write `projectFileIds` optimistically and a project-details refetch re-hydrates to confirm; hydrate must upsert records without clobbering an unreconciled `temp_id` row (`reconcile` owns the temp→server swap). (2) **retention/dangling refs are bounded and safe** — the store holds the session's whole library (in-memory, wiped on restart) and `useFilesByIds` filters missing ids, so a DELETING-skew reference simply doesn't render (explicit `gc()`/LRU is optional, later). (The `File().upload` → `createUploadTask` transport migration for a real `cancel()` is the right call regardless of approach.)

**SoC refinement (post-design, owner-driven):** the chosen C is realized as **a store + a React context, split by concern** — `userFileStore` (zustand engine: files + target-tagged tasks + reconcile; async/high-frequency) and `ComposerDraftContext` (`{text, fileIds}` refs; synchronous UI state, behind `useComposerDraft`). The single-store option was rejected (its only advantage, atomic consume+GC, is moot since every mutation is a synchronous write). Context — not a store — was chosen for the draft because it's small, synchronous, React-written; it stays behind `useComposerDraft` so a later promotion to a store (e.g. for draft-survives-app-kill, or if a keystroke re-rendering the chip strip ever matters) is a contained change with zero component edits. See `03-detailed-design.md` for the final shape.
