# Mobile file-upload module restructure — Implementation Plan

## Issues to Address

The mobile file layer is two forked systems: PR 7's `useProjectFiles` keeps optimistic uploads in a **project-keyed** zustand store (`state/uploadStore.ts`, `byProject`), and PR 8's `useMessageAttachments` keeps a **per-conversation local-state draft** because the project-keyed store can't represent a project-less attachment. They duplicate upload/poll/reconcile logic, and neither survives navigation cleanly. This rework replaces both with **one normalized file cache + a React context, split by concern** (Approach C — single store SSOT; Query fetches-and-hydrates, never renders): a **`userFileStore`** zustand engine whose `filesById` is the **single source of truth for EVERY file record** — uploads, draft attachments, project files, AND recent/library files (each a `FileRecord` keyed by id; each upload is a first-class `tasksById` task tagged with its `target`; async/high-frequency → a store) and a **`ComposerDraftContext`** (`Record<draftKey, { text, fileIds }>`) holding the per-conversation composer draft as text + file *references* (synchronous React UI state → a context, behind `useComposerDraft`). Projects, drafts, and the recent picker hold **only id-references** (`projectFileIds` / draft `fileIds` / `recentIds`) that resolve against the store via `useFilesByIds`. Text and attachments persist across navigation and clear together on send; the two coordinate only via synchronous writes in the `useComposerDraft` hook (no async, no refcount GC). **TanStack Query is demoted to a fetch-and-hydrate LOADER**: it still fetches (retry / dedup / focus-refetch / lazy `enabled`), but every success HYDRATES the store and is **NEVER rendered for files**. One source, several loaders → the dual-source drift that sank the original Approach C is eliminated by the rule "**Query never renders files**".

Outcomes: an upload survives navigation (and completes into the file library instead of being canceled); the composer draft (text + attachments) survives navigation; progress never re-renders the whole file list (task-isolated + atomic selectors); one reconcile reducer (server wins) replaces two forks; and the three deferred P2 bugs — plus the text-input's own copy of the clear-on-bailed-send bug — are fixed structurally. A `UploadTransport` seam + server-as-truth reconciliation keep a future background-upload feature a bounded add. Fixes: **BUG1** clear-on-failed-submit (`ChatSurface.tsx:107-111` + `useChatController.ts:217`), **BUG2** `partitionBySize` null/0 = unlimited (`lib/files.ts`), **BUG3** conversation-keyed async guard → per-run epoch (`useMessageAttachments.ts`), plus the **BearerImage** cleanup rider (`AttachmentImage`/`AgentImage`).

## Important Notes

- **The persistent overlay is the central constraint.** `ChatSurface` mounts once in `app/(app)/_layout.tsx:13` over the `<Stack>` and **morphs** without remounting (`ChatSurfaceContent`, route files render null via `SurfaceRoute`). So any per-conversation state must live outside React (the store) and be isolated by `draftKey`. This is why a store, not context, is mandatory (`mobile/CLAUDE.md`; best-practices: adamhinckley zustand-vs-context).
- **One store is the SSOT for every file record; Query fetches-and-hydrates only, never renders.** `userFileStore.filesById` holds uploads, draft attachments, project files, and recent/library files alike; Query (`QUERY_KEYS.userProject`, `recent-files`) keeps its fetch mechanics but `onSuccess`/`select` hydrates the store and components **never read Query for file rendering** (non-file project-details fields like name/personas may still render from Query). There is no dual-source drift because **nothing renders from Query for files**. **Offline has NO regression:** `projects` and `recent-files` are ALREADY PII-excluded from Query's MMKV persistence (`query/client.ts`), so committed/recent files were never on disk — moving them to an in-memory store loses nothing and the future `persist` slice is identical.
- **Backend evidence the normalized cache mirrors the server's own shape:** `GET /user/files/recent` returns the user's **FULL library** — no LIMIT, no time-bound WHERE, excludes only FAILED/DELETING, `ORDER BY last_accessed_at DESC` (`backend/onyx/server/manage/users.py:1305`); "recent" = the whole file pool, just sorted. A project file is a **pure many-to-many association** (`Project__UserFile` join, `backend/onyx/db/models.py:5069`) — link/unlink add/remove a join row, and a project file is the SAME standalone `UserFile` by id, so project files are a **subset of the same pool** as recent. `UserFileSnapshot.project_id` is hardcoded `None` (`projects/models.py:38`) — the backend itself models **file = identity, membership = association**, so a normalized client cache mirrors the backend (voiding the original C's speculative-generality objection). Web's `ProjectsContext` is the **anti-pattern**, not a model: 5 parallel arrays of full file objects (recentFiles / allRecentFiles / allCurrentProjectFiles / currentMessageFiles / currentProjectDetails.files) hand-reconciled (a 3s poll writes into 3 of them; a 23-dep useMemo splices link/unlink/delete) — the normalized store replaces exactly that duplication.
- **Verified backend facts (do not re-derive):** upload response echoes `temp_id` per file (`UserFileSnapshot.temp_id` via `id_to_temp_id`, `backend/onyx/server/features/projects/models.py:36`) → reconcile-by-`temp_id` reliable. `/user/files/recent` does **not** echo `temp_id` (`backend/onyx/server/manage/users.py:1266`) → dropped-response orphan uploads are surfaced as **retryable failed tasks**, never auto-matched; that same endpoint returns the **full library** (no LIMIT), so `hydrateRecent` seeds the whole pool. `/statuses` keyed by `file.id`, returns `list[UserFileSnapshot]`. Backend resolves an unset max-upload to `min(DEFAULT=100, MAX_ALLOWED=250)=100 MB` (`app_configs.py:67`, `settings/store.py:78`) → client `DEFAULT_MAX_UPLOAD_MB=100`.
- **Reuse, don't rebuild:** pure helpers `lib/files.ts` (`buildOptimisticFile`/`partitionBySize`/`attachmentStatusLabel`/`isFailedFile`/`isImageName`), transport `api/files/upload.ts` (`buildFileKey`/`temp_id_map`/bearer), poller/link/unlink `api/files/files.ts`, pickers `api/files/pickers.ts`, descriptor mapping `chat/fileDescriptors.ts`, contracts `chat/contracts/projects.ts`.
- **Zustand discipline (best-practices):** plain `Record` not `Map`; atomic per-key selectors; never return a fresh object/array from a selector; progress on the task (not the record) so a tick doesn't churn the list; `useShallow` for the id→record list; `subscribeWithSelector` enabled for a future imperative progress bar. Epoch guard (monotonic token), **not** debounce, is the only correctness fix for late async writes (frontendatlas).
- **`onAccepted` timing is load-bearing:** fire after the synchronous active-run guard and **before** the `createChatSession` `await` — guaranteed-committed (fixes both faces of BUG1) yet optimistically snappy.
- **`createUploadTask` migration** (from `File().upload`, for a real `cancel()`) changes the response/error surface — re-validate the non-2xx + JSON-shape guards in `api/files/upload.ts`.
- **Mobile conventions** (`mobile/CLAUDE.md`): NativeWind spacing is pixels; render text via `@/components/ui/text`; hook-per-file; jest-expo tests in `__tests__/`; `@onyx-ai/shared` untouched (chat layer stays mobile-native).

## Implementation Strategy

Ordered so each step is a coherent, independently-reviewable change; Phase 5 bundles them into PRs.

1. **Extract `BearerImage` (independent, no store dependency).** Add `components/ui/BearerImage.tsx` owning the `expo-image` + `useAuthToken` bearer + `cachePolicy="none"` + neutral placeholder invariant; collapse `components/chat/AttachmentImage.tsx` and `components/avatars/AgentImage.tsx` to one-line wrappers. (Rider — ships first, unblocks nothing, de-risks the diff.)

2. **BUG2 fallback helper.** In `lib/files.ts` add `DEFAULT_MAX_UPLOAD_MB = 100` + `resolveMaxUploadMb(mb)`; route `partitionBySize` through it (finite fallback replaces the inline null/0 check).

3. **Transport seam.** Add `api/files/transport.ts` (`UploadTransport` interface + `foregroundTransport` + `getUploadTransport`/`configureUploadTransport`). Migrate `api/files/upload.ts` to a `createUploadTask`-based cancelable uploader that the foreground transport wraps; keep the manual bearer, `temp_id_map`, and non-2xx/JSON guards.

4. **The engine store + the draft context.** (a) `state/userFileStore.ts` (rewrite of `uploadStore.ts`): `filesById` (now the SSOT for EVERY file record) / `tasksById`(target-tagged) / `projectFileIds: Record<number, string[]>` (a project's ordered committed membership) / `recentIds: string[]` (the ordered recent/library list) / `errorsByTarget` / `epochCounter` + `runtimeHandles`; `UserFileActions` (`beginUpload`, `setProgress`, `failTask`, the one `reconcile` reducer — UNCHANGED: match by temp_id else id, server wins — `registerExisting` (single recent-attach for a draft), `hydrateProject(projectId, files)` (upsert records + set `projectFileIds`), `hydrateRecent(files)` (upsert records + set `recentIds`), `removeFile`, target-error actions) + atomic selectors (`useUploadProgress`/`useFilesByIds`/`useTargetUploadIds`/`useTargetErrors`); `subscribeWithSelector`. (b) `components/chat/ComposerDraftProvider.tsx` (new React context): `Record<draftKey,{text,fileIds}>` via reducer + `setText`/`addFiles`/`removeFile`/`consume`/`consumeAttachments`, mounted above `ChatSurface`, consumed only through `useComposerDraft`. The engine never imports the draft context or Query.

5. **Core engine `hooks/useUpload.ts`.** Size pre-check (`resolveMaxUploadMb`) → `beginUpload` → `getUploadTransport().upload(...)` (progress → `setProgress` with the captured epoch; cancel handle → `runtimeHandles`) → `await result` → `reconcile(user_files, epoch)` + rejected → `failTask`. A **project** upload appends the optimistic id to `projectFileIds[projectId]` and refetches project-details to confirm server membership (re-hydrate) — there is **no Query hand-off** (no `invalidateProject(); removeFile(clientId)`): the record STAYS in the store. Returns the optimistic `clientId[]`; no React state of its own.

6. **`UploadReconciler` (`components/chat/UploadReconciler.tsx`) + mount — the SINGLE poller.** 3s `/statuses` poll over **every** `isServerProcessingStatus` file in `filesById` (project + draft + recent, any surface) that has a server id → `reconcile`; an `AppState "active"` listener firing the same poll (guarded to when processing files exist, debounced); a temp-id record whose task is gone stays a retryable `failed` task. There is **NO per-surface status poll** — no `usePendingStatusPoll`, no `setQueryData` status-patcher; the one reconciler over the store serves all surfaces. Mount `<UploadReconciler/>` in `app/(app)/_layout.tsx` beside `<ChatSurface/>`.

7. **Draft lens `hooks/useComposerDraft.ts` (rename of `useMessageAttachments.ts`).** `text`/`setText` (via `useDraftText`/`setDraftText`), attachment surface over `useDraftIds`→`useFilesByIds`, `descriptors`/`hasBlockingFiles`, upload/attach/remove via `useUpload` `{kind:"draft",draftKey}`, `consume`/`consumeAttachments`/`dismissErrors`. Delete the old local-state/keyRef/poll engine.

8. **Project + recent lenses (rewrite).** `hooks/useProjectFiles.ts`: read `projectFileIds[projectId]` → `useFilesByIds` for the committed list (the store, NOT a Query-rendered `committedFiles`), prepend in-flight tasks (`useTargetUploadIds`→`useFilesByIds`); uploads via `useUpload` `{kind:"project",projectId}`; `linkFileToProject`/`unlinkFileFromProject` update `projectFileIds` **optimistically** then refetch project-details to re-hydrate membership (server-authoritative); NO Query-rendered committed list and NO per-surface poll. `hooks/useRecentFiles.ts`: fetch `/recent` → `hydrateRecent`; the recent **picker reads `recentIds`** from the store (id-refs → `useFilesByIds`), NOT Query results; NO per-surface poll. `useProjectDetails` becomes a loader — its fetch `onSuccess` calls `hydrateProject`, and only non-file fields (name/personas) render from Query. Public shape preserved (drop `progressById`).

9. **Controller `hooks/useChatController.ts` (BUG1).** Remove `input`/`setInput`; `submit(text, files?, onAccepted?)`; delete the `setInput("")` at `:217`; invoke `onAccepted?.()` at the guaranteed-committed point (after the active-run guard, before `createChatSession`).

10. **Wire `ChatSurface.tsx` + composer consumers.** `const draft = useComposerDraft(draftKey)`; `InputBar value={draft.text} onChangeText={draft.setText}`; delete the `setInput("")`-on-switch effect; `sendWithAttachments(message?)` → `submit((message ?? draft.text).trim(), draft.descriptors, message == null ? draft.consume : draft.consumeAttachments)`. Update `FileCard.tsx` (read `useUploadProgress`; drop `progress` prop + read task error), `InputBar.tsx`/`ProjectContextPanel.tsx` (drop `progressById` prop pass).

## Plan Challenge Results

### 1. Extendability & Scalability: PASS (one mitigated concern)
New surfaces (file-library screen, background transport) and new file contexts plug into existing seams (`UploadTransport`, the `UploadTarget` union + a link map, atomic selectors) — no rewrite. With the SSOT this is stronger still: a **file-library screen is now literally a new selector over the same store** (`filesById` already holds the whole library via `hydrateRecent`) plus a loader — no new cache, no reconciliation. The one honest concern: `setProgress` spreads the `filesById`/`tasksById` `Record` each tick (O(#files)); negligible for a chat composer / project panel's realistic handful of concurrent uploads, and the documented escape hatch is a `subscribeWithSelector` transient/imperative progress bar. No hardcoded limits except `DEFAULT_MAX_UPLOAD_MB=100`, which mirrors the verified backend default.

### 2. Fragility: PASS (brittle points identified + hardened)
The original C's **dual-source drift** fear — the reason it was first rejected — is **REMOVED by construction**: with the rule "**Query never renders files**", there is only ever one thing on screen (the store), so two homes can't disagree. Two real ordering/coupling dependencies remain, both documented and test-covered: (a) the **epoch guard** must be threaded through every guarded write — hardened by keeping the guarded actions callable only from `useUpload`/`UploadReconciler` (not exported for ad-hoc use); (b) **`onAccepted` timing** (after the active-run guard, before the `createChatSession` await) — hardened by a `useChatController` test asserting it fires only on a committed send. Plus: `reconcile` atomic in one `set`. **Retention:** the store now holds the session's whole library (bounded by library size, in-memory, wiped on restart); dangling refs are **safe — `useFilesByIds` filters missing ids** (a DELETING-skew id simply doesn't render), and an LRU is an optional later add. **Membership stays server-authoritative** via `hydrateProject` + optimistic add/remove + refetch. No un-mitigated single point of failure.

### 3. Industry Standard: VERIFIED (web)
- **Zustand atomic selectors / never return a fresh object-array / `useShallow` / stable module-const selectors** — confirmed as the canonical 2025 performance pattern ([pmndrs/zustand](https://github.com/pmndrs/zustand), [dev.to: Avoid performance issues](https://dev.to/devgrana/avoid-performance-issues-when-using-zustand-12ee)).
- **Normalized `byId` entity store for an entity shown across multiple screens, single source of truth, partial-merge updates (server wins)** — confirmed standard ([Redux: Updating Normalized Data](https://redux.js.org/usage/structuring-reducers/updating-normalized-data), [Advanced Redux Entity Normalization](https://medium.com/@dcousineau/advanced-redux-entity-normalization-f5f1fe2aefc5)).
- **Store-outside-React over Context for detached-async / survives-unmount / high-frequency state** — confirmed ([State Management in 2025](https://dev.to/hijazi313/state-management-in-2025-when-to-use-context-redux-zustand-or-jotai-2d2k)).

### 4. Fact Check: PASS
- **Epoch/request-id guard, not debounce** — the plan's BUG3 fix is explicitly web-confirmed: "debouncing reduces request count but does not prevent out-of-order responses… insufficient for correctness"; the incrementing request-id/counter is the actual guard ([frontendatlas: Async Race Conditions](https://frontendatlas.com/javascript/trivia/js-async-race-conditions)). The one documented request-id edge case (reuse of an id across two requests for the same resource) does **not** apply here: epochs are per-run monotonic (never reused) and identity is per-file `clientId`, with `removeFile` cancellation on top.
- **Backend facts** (`temp_id` echo on upload but not `/recent`; `/statuses` keyed by `file.id`; 100 MB default) were verified against the actual source (`projects/models.py:36`, `manage/users.py:1266`, `app_configs.py:67`) — stronger than a web claim.
- **expo-file-system `createUploadTask`/`sessionType` background caveats** verified in Phase 1 against the official Expo docs.

### 5. Maintainability: PASS
The mental model is small — **one cache for all files** (not two homes to keep in sync), a draft context, one reconcile reducer, hydrate-on-load loaders, and one transport interface — and it **replaces two forked systems plus Query's file-rendering role with a single SSOT**, a net simplification a newcomer can hold in 15 minutes. Every file lives in exactly one place; loaders only feed it; views only select from it. It follows the app's existing conventions (zustand for `chatSessionStore`/`session`; context for `SidebarProvider`). The renames (`uploadStore→userFileStore`, `useMessageAttachments→useComposerDraft`) make scope obvious. Where to change in 6 months is clear (add a `UploadTarget` variant + selector). The store/context split is the SoC win: the engine has no composer concepts and is reusable (file-library screen, background upload), and the draft stays behind `useComposerDraft` so promoting it to a store later is contained.

### 6. Patch vs. Fix: PROPER FIX
This is the opposite of a patch — it is a root-cause restructure. Each bug is eliminated by construction, not worked around: **BUG1** the draft lives in the store and survives `submit` (not a try/catch), cleared only on an accepted send; **BUG2** a finite fallback replaces the null=unlimited logic at its single source (not a bumped limit); **BUG3** a monotonic epoch guard — the plan *explicitly rejects* debounce (the web-confirmed non-fix) — replaces the fragile conversation-keyed guard. The underlying root cause (two forked, ephemeral file layers keyed by the wrong thing) is removed entirely. **No patch-vs-fix decision needed from the user.**

**Verdict: all six checks pass.** The design was revised from Approach B to **Approach C — a single normalized store SSOT where Query fetches-and-hydrates but never renders files** (backend evidence: recent = the full library, a project file is a `Project__UserFile` join of the same `UserFile`, `project_id` hardcoded `None`; the original C's dual-source-drift objection is voided by "Query never renders files" and offline shows no regression since projects+recent were already MMKV-excluded). Two concerns (progress-spread at extreme scale; epoch/`onAccepted` ordering discipline) are pre-mitigated in the plan and carried into the PR roadmap's checkpoints.
